# Part 1 — K3s Cluster with Vagrant (Server + Worker)

This document explains **everything in Part 1**: what is installed, why each program is used, how the cluster is bootstrapped, how files work together, and how to test/debug during evaluation.

---

## 1) Goal of Part 1

Create a small Kubernetes cluster using **2 virtual machines**:

- `aliS` → K3s **Server** (control plane)
- `aliSW` → K3s **Agent** (worker)

Both machines are created with Vagrant + VirtualBox and communicate over a private network.

---

## 2) Programs and What They Do

### VirtualBox
- Hypervisor that actually runs virtual machines.
- Provides CPU/RAM/network virtualization.
- In this project, VirtualBox hosts both Linux VMs.

### Vagrant
- VM automation tool.
- Reads `Vagrantfile` and creates/configures machines automatically.
- Runs shell provisioning scripts after the VM boots.
- Provides synced folder (`/vagrant`) between host and guest.

### Debian Bookworm box (`debian/bookworm64`)
- Base OS image used by Vagrant.
- Chosen for lightweight and stable behavior.

### K3s
- Lightweight Kubernetes distribution by Rancher.
- Server mode runs Kubernetes API/control plane.
- Agent mode joins server and runs workloads.

### `curl`
- Used to download/install K3s with official install script (`https://get.k3s.io`).
- Also used by worker script to probe K3s API readiness.

---

## 3) Architecture

```
Host Machine
└── VirtualBox
    ├── VM1: aliS   (192.168.56.110)
    │   └── K3s Server (control plane)
    └── VM2: aliSW  (192.168.56.111)
        └── K3s Agent  (worker)
```

Network details:
- `eth0` = NAT (internet access)
- `eth1` = private host-only network (VM-to-VM communication)

K3s overlay networking (flannel) is pinned to `eth1` to avoid interface confusion.

---

## 4) Directory and File Roles

```
p1/
├── Vagrantfile
├── node-token              # generated during provisioning
└── scripts/
    ├── server_setup.sh
    └── worker_setup.sh
```

### `Vagrantfile`
Defines both VMs:
- hostnames
- private IP addresses
- CPU/RAM
- provisioning script paths

### `scripts/server_setup.sh`
On `aliS`:
1. Installs prerequisites
2. Installs K3s in server mode
3. Waits until node token is available
4. Shares token to `/vagrant/node-token` atomically

### `scripts/worker_setup.sh`
On `aliSW`:
1. Installs prerequisites
2. Waits for non-empty `/vagrant/node-token`
3. Waits for K3s API (`192.168.56.110:6443`) to become reachable
4. Installs K3s in agent mode and joins the server

### `node-token`
- Secret token generated by K3s server.
- Required for secure worker join.
- Shared through Vagrant synced folder (`/vagrant`).

## 5) Why These K3s Flags Are Used

### Server flags
- `--node-ip=192.168.56.110`
  - Advertises correct node IP to cluster.
- `--bind-address=192.168.56.110`
  - API server listens on private network interface.
- `--flannel-iface=eth1`
  - Forces flannel to use host-only/private NIC, not NAT NIC.
- `--write-kubeconfig-mode=644`
  - Allows non-root `kubectl` reads inside VM.

### Worker flags
- `K3S_URL=https://192.168.56.110:6443`
  - Server API endpoint for registration.
- `K3S_TOKEN=<node-token>`
  - Authentication to join cluster.
- `--node-ip=192.168.56.111`
  - Worker’s cluster node IP.
- `--flannel-iface=eth1`
  - Ensures consistent inter-node overlay networking.

---

## 6) Provisioning Flow (Step by Step)

1. `vagrant up` starts both VMs.
2. `aliS` runs `server_setup.sh`:
   - installs K3s server
   - creates `node-token`
   - copies token to synced folder
3. `aliSW` runs `worker_setup.sh`:
   - waits for non-empty token
   - waits for API readiness
   - installs K3s agent and joins server
4. Cluster reaches two-node `Ready` state.

## 7) Commands You Should Know

### Start from scratch
```bash
cd p1
vagrant destroy -f
rm -f node-token
vagrant up
```

### SSH into server
```bash
vagrant ssh aliS
```

### Verify cluster
```bash
kubectl get nodes -o wide
```

Expected shape:
- `aliS` with control-plane role
- `aliSW` with worker role (`<none>`)
- both in `Ready`

### Check K3s services
```bash
sudo systemctl status k3s
sudo systemctl status k3s-agent
```

### Follow live logs
```bash
# on aliS
sudo journalctl -u k3s -f

# on aliSW
sudo journalctl -u k3s-agent -f
```

## 8) Why Worker Can “Stuck” and How This Setup Prevents It

Common race conditions:

1. **Token file exists but is empty/partial**
   - Fixed by checking `-s` (non-empty) and atomic copy on server.

2. **Worker starts before API is truly reachable**
   - Fixed by explicit API readiness loop using `curl`.

3. **Wrong network interface selected automatically**
   - Fixed by forcing flannel to `eth1`.

4. **Old residual state from previous failed run**
   - Use `vagrant destroy -f` and remove `node-token`, then `vagrant up`.

## 10) Troubleshooting Guide

### A) Worker keeps waiting forever
- Check token file on worker:
  ```bash
  ls -l /vagrant/node-token
  cat /vagrant/node-token
  ```
- Check API reachability from worker:
  ```bash
  curl -k https://192.168.56.110:6443/
  ```

### B) Worker service fails repeatedly
```bash
sudo systemctl status k3s-agent --no-pager -l
sudo journalctl -u k3s-agent -n 200 --no-pager
```

### C) Server appears up but cluster unstable
```bash
sudo systemctl status k3s --no-pager -l
sudo journalctl -u k3s -n 200 --no-pager
kubectl get pods -A -o wide
```

### D) Full clean reset
```bash
cd p1
vagrant destroy -f
rm -f node-token
vagrant up
```
